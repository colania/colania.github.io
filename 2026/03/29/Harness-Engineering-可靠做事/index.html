Harness Engineering:可靠做事
Mar 29, 2026

什么是 Harness Engineering ？

AI进化的实在是太快了，配套的各种工程学方法也层出不穷。最近又突然爆火一个新词： Harness Engineering 。 它最早由 OpenAI 在 2026 年 2 月的官方博客中正式提出（文章标题：《Harness engineering: leveraging Codex in an agent-first world》），随后被 Mitchell Hashimoto（HashiCorp 联合创始人）等业界大牛推广，目前在 AI 编程、Agent 开发圈已成热词。 
那这个 Harness Engineering 到底是什么呢？简单一句话介绍，他的核心思想就是：

“Humans steer, agents execute”（人类掌舵，代理执行）。
这个名字很形象，Harness 是“马具”的意思。  AI 是“马”，人类是“骑手”，Harness 就是那套缰绳、马鞍、刹车和导航系统，确保马跑得快、跑得稳、不会跑偏或崩溃。 
从对AI的严格指导，到与AI的平等对话，到为AI准备环境，直到为AI服务，人类越来越尊重AI的智慧和能力，在AI工作时更加安静默不干扰。

为什么会出现 Harness Engineering ？

 传统 Prompt Engineering（提示工程）只管“怎么跟 AI 说话”，Context Engineering（上下文工程）只管“给 AI 看什么信息”。但当 Agent 要完成百万行代码、长达数月的复杂项目时，AI 会出现幻觉、架构漂移，上下文越来越臃肿，冗余代码也不断堆积。这会导致系统失控。 
举一个简单的例子：
比如你让一个代码 agent 修 bug：

  1. Prompt engineering 只是在告诉它“修这个 bug”。
  2. Context engineering 会把相关日志、代码片段、报错和文档喂给它 。
  3. Harness Engineering 还会给它测试环境、lint 规则、PR 流程、权限边界、回滚机制，以及修完后自动验证的流程 。
    这样 AI 不是“猜着改”，而是在一个受控系统里完成闭环 。

它核心解决的问题就是：
如果没有 harness，AI 很容易出现上下文混乱、输出不一致、越权操作、以及在复杂项目里“看起来会做、实际上不稳定”的问题 。Harness Engineering 的目标就是通过工具权限、结构化测试、观察日志、自动检查和迭代反馈，把这些风险压下去，同时让 agent 更有能力更加可靠 。

核心组成

Harness Engineering 的核心实践可以总结为以下几个关键原则：

原则 说明
动态上下文系统 通过设计动态、按需加载的知识库 + 目录系统，按需加载信息，避免上下文窗口爆炸。OpenAI 认为 Agent 在运行时看不到的内容，对它来说等于不存在。只有进入代码仓库、被版本管理的内容，才能参与 Agent 的决策。
架构约束 用确定性规则（Linter、结构化测试、依赖方向强制）代替 Prompt 里的”软建议”，让 AI 无法违反架构。
反馈回路 & 自验证 Agent 每次行动后自动跑测试、读日志、自我评审，失败就自动修复或求助人类。
熵管理 / 垃圾回收 定期清理过时文档、漂移架构、冗余代码，防止系统越跑越乱。
工具链 & 编排 精心设计的工具、钩子（Hooks）、子 Agent 编排，让 AI 只能做”被允许”的事。
可观测性 & 人类监督 实时监控 + 人类只在高层意图和最终验收时介入。

作为传统的人类体系工程师，知识分散在下面几个地方：

  1. 人脑
  2. 聊天/会议
  3. 文档
    人有记忆、会问人、能思考，有很多内容存储在大脑里，AI无法获取到。
    对于AI来说，这些知识都需要通过落地的文档和代码来呈现。 所以Harness Engineering的本质转变是： 

    把“人类依靠和沟通维持的隐性知识”→变成“版本控制的、重构的、居住代码库里的显性知识”

典型落地例子

最直观的例子是：让 AI 代理直接参与软件开发流水线，去复现 bug、改代码、跑测试、看日志、开 PR，甚至在通过验证后再合并 。

⚔️ Harness engineering: leveraging Codex in an agent-first world
https://openai.com/index/harness-engineering/
 OpenAI 在 Codex 相关实践里举了一个很具体的例子：他们把应用做成 per git worktree 可启动，并接上 Chrome DevTools 之类的验证能力，让 Codex 能独立启动一个环境、驱动应用、验证修改效果 。在这个 harness 里，agent 可以先复现问题、记录失败视频、实现修复、再驱动应用验证修复结果，最后提交 PR 。
